/**
 * P1. EventBus 是 EVENTS Sheet 唯一的写入者
 *     任何模块都不允许直接写 Task / Reminder / Inventory / AI输出 的状态。
 *
 * P2. 真相是不可变、只追加的（Truth is immutable, append-only）
 *     EVENTS Sheet 只增不改不删。修正错误也是发一条新事件，不是改旧记录。
 *
 * P3. 状态永远是从事件派生出来的（State is derived from Events）
 *     任何「当前任务列表」「当前库存」「当前指标」都是重放 EVENTS 算出来的，
 *     不允许有第二套独立的持久化状态表。
 *
 * P3.1【澄清】「派生视图」表是允许直接写的，但只能整行覆写算出来的结果
 *     Tasks Sheet / Inventory Sheet / ActiveTasks Sheet / ArchiveTasks Sheet
 *     属于「派生视图」：方便人看、方便 Rider OS 指令直接查询，但它们的每一行
 *     内容都必须 100% 是事件派生的，绝不允许有人在 Sheet 上手动改字段后这个
 *     改动就直接生效——任何想要的改动都必须先变成一个事件（EventBus.publish），
 *     再让派生视图覆写过去。
 *     技术实现：05_SheetUtils.gs 的 upsertRowByKey_() / deleteRowByKey_() /
 *     batchUpsertRowsByKey_() 就是专门干这件事的，只允许 Tasks/Inventory/
 *     ActiveTasks/ArchiveTasks 这类派生视图调用，不允许碰 EVENTS 表。
 *
 * P4. AI 只建议，绝不执行（AI only suggests, never executes）
 *     AI 模块产出的是建议/优先级/分析，真正的状态变更必须经过用户确认 →
 *     再走 EventBus.publish()。
 *
 *     【V4 AI-Native Advisory Architecture 的边界】
 *     V4（见 P6.1 和 P6.2）扩展了 AI 在"理解-推理-规划-建议"层的能力，
 *     但不改变 P4：AI 提供 Decision Proposal（决策提案），Execution 永远
 *     发生在 User Confirmation（用户确认）之后，再走 EventBus.publish()。
 *     这条不变。
 *
 * P5. Output.gs 是唯一的 Telegram 发送者
 *     任何模块要给用户发消息，必须经过 Output.gs，不允许自己调 Telegram API。
 *
 * P6. 不在真实事件数据积累之前过度设计
 *     新的 OS 级别模块（独立子系统）需要先有真实使用数据支撑，
 *     不要为了「架构完整」而提前造模块。
 *
 * P6.1【澄清，2026-06-25 新增，2026-06-29 更新，2026-07-03 再更新】
 *     外部AI（非Claude）生成的提案的评审记录：
 *
 *     ❌ 已驳回（不进入架构）：
 *       V5 (Unified OS / CrossSystemBus) — V5 的 EntityModel 想把 Task/Rider/
 *         Inventory/Finance 统一成一个 Entity 系统，跟已确认的"Rider OS 是独立
 *         项目（独立GAS+独立Spreadsheet+独立Telegram chat），只接受 RiderConnector
 *         单向只读同步"这条边界直接冲突，驳回。
 *       V6 (Autonomous Agent / AutoScheduler / ProactiveTrigger) — "自动重排/
 *         自动取消/自动执行"的部分跟 P4 冲突（AI 只建议，绝不执行），驳回。
 *       V7 (Digital Twin / LifeSimulationEngine / AutonomyController) — V6 的
 *         极端版本，同时违反本条 P6，驳回。
 *
 *     ✅ 已采纳：
 *       V3 (Task Intelligence) — 已实现。
 *       V4 (Memory+Behavior+Suggestion Engines) — 已实现（93/94/95/96）。
 *
 *     ✅ 2026-06-29 重新评审后采纳（见 P6.2）：
 *       外部AI提案中的 "Unified OS Architecture V3 / V4 AI-Native" 经重新评审，
 *       核心实质分两部分：
 *       (a) 数据合并部分 → 仍然驳回（同 V5，违反 Rider OS 独立性边界）
 *       (b) 协议统一部分 → 采纳，但重命名为"Unified Domain Protocol (UDP)"
 *           并收录进 P6.2，不改 P4，不改 Rider OS 边界。
 *       AI-Native 执行自动化部分 → 驳回（违反 P4）。
 *       AI-Native 顾问架构部分 → 采纳，但命名为
 *           "V4 AI-Native Advisory Architecture (V4-AAA)"，
 *           重申：AI 的职责是 理解→推理→规划→提建议，Execution 仍须用户确认后
 *           才走 EventBus.publish()，不改 P4。
 *
 *     ✅ 2026-07-03 重新评审后采纳（见 二/平台架构 D1-D8，P6.2 升级）：
 *       外部AI提案 "JARVIS Platform V4/V5"（Core+Domain OS 拓扑重组 +
 *       IMIOR Pattern + Domain OS Blueprint + UDP 三层标准 + Engine
 *       命名规范 + Voice OS）经两轮评审（Carson 对 Claude 的初版又提出
 *       了修正意见），最终采纳版本：
 *       (a) Core+Domain OS 拓扑 → 采纳为目标拓扑，见 D2。其中 Procurement
 *           OS（原 Shopping OS）和 Inventory OS 改判为 External System
 *           （不是 Core 治理的 Domain OS）——因为这两个系统已经在独立
 *           开发中，Core 这边只负责建 Connector，不再规划"共享 Identity
 *           Registry"式的深度集成（这点否决了 00_Project_State.gs 原
 *           下一步第16条的方案，已同步修正，见 D7）。Task/Reminder 确认
 *           不属于 Core，分别归 Productivity OS / Reminder OS（IMIOR 只
 *           有五个支柱，Task/Reminder 是业务领域，不是通用能力）。
 *       (b) Rider OS 物理边界 → 不受影响，继续独立 GAS + 独立
 *           Spreadsheet + 独立 Telegram chat，见 D2。
 *       (c) IMIOR Pattern（Core 自身架构）→ 采纳，见 D3。
 *       (d) Domain OS Blueprint（Foundation/Runtime/Intelligence）→
 *           采纳，见 D6。
 *       (e) UDP 从两层再升级为三层标准 → 采纳：Level 1 Platform
 *           Standard（新增，见 D1-D5，不属于 UDP 本身）+ Level 2 Domain
 *           OS Standard / Level 3 Engine Standard（= UDP，见 P6.2）。
 *           Level 2 的 Execution/Event 关系修正为标准 Event Sourcing
 *           的 Command → immutable Event（Execution 发出指令，Event 才
 *           是真正的写入事实），Confirmation 独立成一个具名层（不再是
 *           Decision 层的括号备注）。Engine 命名规范采纳，见 D8。
 *       (f) Voice → 不采纳为独立 OS，进一步泛化为 Core.Intent 的
 *           "Input Adapter"概念：Telegram 文字/Voice/未来的 Image/PDF/
 *           Email 都是同一套 Intent Router 的不同输入口，不因为多一种
 *           输入方式就多建一个 OS，见 D3。
 *       本条不改变 P1-P5，不改变 Rider OS 物理边界。
 *
 *     如果将来真的要做"自动执行"，必须先在这里明确修改/废止 P4，
 *     不能在某次代码生成里悄悄绕过。
 *
 * P6.2【2026-06-29 新增，2026-07-03 升级为三层标准里的 Level 2+3】
 *     Unified Domain Protocol (UDP)
 *
 *     UDP 现在特指 Level 2（Domain OS Standard）+ Level 3（Engine
 *     Standard）——"协议"这个词专指行为/生命周期标准。平台整体的组成
 *     声明（IMIOR Core / Domain OS 花名册 / Shared Services /
 *     Cross-domain Bridge）是 Level 1 Platform Standard，见「二、平台
 *     架构」D1-D5，不属于 UDP 本身，UDP 只是 Level 1 之下的一部分。
 *
 *     ── Level 2：Domain OS Standard（整个 Domain OS 必须遵循）───────────
 *
 *       Request → Planner → Decision → Confirmation → Execution → Event
 *       → Projection → Insights → Bridge
 *
 *       各层职责：
 *         Request       输入捕获（Telegram 消息/回调/定时触发器等）
 *         Planner       结构化整理输入，识别意图/参数
 *         Decision      选择与优先级判断，产出 Decision Proposal（决策
 *                       提案）
 *         Confirmation  见下方强制条款——不是所有场景都需要，但需要时
 *                       不可省略
 *         Execution     Execution Layer issues commands that produce
 *                       immutable Events —— Execution 层的职责是"发出
 *                       确认后的业务指令"（Command），例如"创建这个
 *                       任务"，Execution 本身不直接写数据
 *         Event         Command 被处理后落下的不可变记录，由
 *                       EventBus.publish() 写入 Events——真正的"写"
 *                       动作发生在这一层，不是 Execution 那一层。
 *                       Execution → Event 是标准 Event Sourcing 里的
 *                       Command → Event 关系：先有确认后的指令
 *                       （Execution），指令被处理后才产生事实
 *                       （Event），两者是先后的两个层，不是同一个
 *                       动作的两种说法
 *         Projection    ProjectionEngine 消费 Event，增量更新 Read Model
 *         Insights      从 Read Model / Events 里做模式分析（对应
 *                       Reflection）
 *         Bridge        跨系统集成，规则见 D5
 *
 *       Confirmation 强制条款（不因升级为三层标准而放松，仍是 P4 的
 *       硬性要求）：
 *         - 如果 Decision 会导致状态变更（即会走到 Execution/Event 这
 *           两层，触发 EventBus.publish()），Confirmation 是必经步骤，
 *           不可省略。
 *         - 如果 Decision 只产出只读结果（不会走到 Execution/Event，
 *           比如一次查询/一次统计展示），则不需要 Confirmation。这类
 *           场景通常根本用不到 Level 2 全链路，应该直接用 Level 3
 *           Engine Standard。
 *
 *       Level 2 的既有边界（沿用原 P6.2，未改变）：
 *         - 现有系统（Rider OS、20/21 模块、V4 引擎层等）不需要为了
 *           UDP 立即重构；如果要接入，走适配器模式（adapter pattern）
 *           逐步迁移，不用一次性重写。新建的 Domain OS / Engine 必须
 *           完整遵循。
 *         - 协议不改变 Domain 之间的物理边界（Rider OS / Procurement
 *           OS / Inventory OS 这类 External System 继续独立，见 D2）。
 *         - 协议不允许跨 Domain 合并数据（只统一协议，不合并 Entity）。
 *         - AI（Personal AI Core）可以在 Planner / Decision 层参与
 *           推理和建议，但任何状态变更仍须遵循 P4，经 Confirmation
 *           后才可发布事件。
 *
 *     ── Level 3：Engine Standard（OS 内部单个 Engine，按需选用）────────
 *
 *       UDP 的 Level 2 是 Domain OS 整体的标准，不是每个 Engine 的
 *       标准。一个 Engine 只要不涉及状态变更，就不需要 Event/
 *       Projection/Bridge 这几层——硬套只会逼着生成没有意义的空函数。
 *
 *       Engine 通用模板：
 *
 *         Input → Validate → Business Logic → Output
 *
 *       典型映射（不是穷举，新 Engine 按同样思路套）：
 *         Priority Engine       Task → Calculate Score → Return Ranking
 *                                （纯计算，不需要 Event/Projection/
 *                                Bridge）
 *         Search Engine         Request → Parse Filter → Query
 *                                Projection → Output（只读 Projection，
 *                                不产生新 Event）
 *         Dashboard Engine      Projection → Aggregate → Render →
 *                                Output
 *         Reminder Rule Engine  Rule → Planner → Reminder Plan → Output
 *         Analytics Engine      Projection → Insights → Output
 *
 *       什么时候一个"Engine"其实需要升级成走完整 Level 2？
 *         判断标准很简单：这个 Engine 会不会自己触发
 *         EventBus.publish()？会 → 它本质上是在做 Execution，必须走
 *         Level 2（含 Confirmation）。不会（只读 Projection 或纯
 *         计算）→ Level 3 足够。
 */

/**
 * D1. JARVIS Unified Standard —— 三层标准总览
 *
 *     2026-07-03 定稿：标准分三层，层级越高越抽象，层级越低越具体。
 *     UDP（Unified Domain Protocol，P6.2）specifically 指 Level 2 +
 *     Level 3——"协议"这个词专指行为/生命周期层面的标准，不包括
 *     Level 1 的拓扑/组成声明。
 *
 *       Level 1 — Platform Standard（整个 JARVIS 平台的组成）
 *         IMIOR Core（见 D3）+ Domain OS 花名册（见 D2）+ Shared
 *         Services（见 D4）+ Cross-domain Bridge（见 D5）
 *         这一层回答的问题是："JARVIS 平台由哪些东西组成？"
 *
 *       Level 2 — Domain OS Standard（= UDP 的一半，见 P6.2）
 *         每一个 Domain OS 整体必须遵循的完整生命周期：
 *         Request → Planner → Decision → Confirmation → Execution →
 *         Event → Projection → Insights → Bridge
 *         这一层回答的问题是："一个 Domain OS 应该怎么跑完一次业务
 *         流程？"
 *
 *       Level 3 — Engine Standard（= UDP 的另一半，见 P6.2）
 *         Domain OS 内部单个 Engine 的轻量模板：
 *         Input → Validate → Business Logic → Output
 *         这一层回答的问题是："OS 内部一个具体的函数模块该怎么组织？"
 *
 *     好处：以后不管加多少个 Domain OS、多少个 Engine，都套同一套三层
 *     骨架，Claude／其他 AI 生成代码时不会走样，人也更容易在几十个模块
 *     里快速定位"这段代码到底是平台级/OS级还是Engine级的东西"。
 *
 * D2. Platform 拓扑
 *
 *     Domain OS（归 JARVIS Platform Blueprint 治理，Core 通过 Level 2
 *     生命周期调度）：
 *       Productivity OS   ← 2026-07-03 已物理拆分为独立 Apps Script 项目
 *                            （原 06/20/13），以 Library 形式被本项目引入
 *                            （Identifier: ProductivityOS）。名字保留
 *                            Productivity OS，不改 Task OS（Carson 确认：
 *                            未来职责会远超过 Task，范围包含 Task/
 *                            Priority/Recurring/Search/Archive/
 *                            Statistics/Dashboard 等 Engine）。部署/接线
 *                            细节见本项目 README.md。
 *       Reminder OS       ← 2026-07-03 已物理拆分为独立 Apps Script 项目
 *                            （原 92），完全独立运作（自己的时间触发器，
 *                            不被本项目当 Library 调用，也不调用本项目），
 *                            只靠读写同一张共享 Spreadsheet 联系。定位：
 *                            全平台共享的时间与通知服务，不是 Productivity
 *                            OS 专属——现在 Productivity OS 用，未来
 *                            Property/Finance/Vehicle OS 的到期提醒也会
 *                            用同一个服务，不会各自重复造一套提醒逻辑
 *                            （跨 OS 接入方式见 D5）。跟 D4 Shared
 *                            Services 不同：Reminder OS 有自己完整的
 *                            业务状态（提醒规则/升级策略/发送历史），
 *                            是一个真正的 Domain OS，只是"客户"是其他
 *                            Domain OS，不是终端用户直接对话的对象。
 *       Property OS       ← 规划中，尚无文件
 *       Calendar OS       ← 规划中，尚无文件
 *       Future（暂不立项，同 P6）：Finance OS / Vehicle OS / Health OS
 *
 *     External System（独立于 JARVIS Platform 之外，Core 只建
 *     Connector，不治理它们的内部架构，不共享 Identity Registry）：
 *       Rider OS          ← 已经是独立 GAS + 独立 Spreadsheet + 独立
 *                            Telegram chat，Core 通过
 *                            80_RiderConnector.gs 单向只读桥接
 *                            （Cross-domain Bridge，见 D5）
 *       Procurement OS（原 Shopping OS）
 *                         ← 2026-07-03 改判为此类：已经在独立开发中，
 *                            Core 这边只负责未来建 Connector，不再规划
 *                            "共享 Identity Registry"式的深度集成（原
 *                            方案作废，见 D7 和 00_Project_State.gs
 *                            下一步第16条）
 *       Inventory OS（V2） ← 2026-07-03 改判为此类：已经在独立开发中，
 *                            同上，只建 Connector。现有
 *                            21_InventoryModule.gs /
 *                            22_InventoryIntentParser.gs（V1）今天不动，
 *                            继续在 Core 内运作，退役时机跟 V2
 *                            Connector 就绪时间挂钩，见 D7
 *
 *     Voice、Keyboard（现在的 Telegram 文字）、未来可能的 Image/PDF/
 *     Email 都不是 OS，是 Core.Intent 的 Input Adapter，见 D3。
 *
 * D3. Personal AI Core = IMIOR Pattern
 *
 *     Core 不再无限长大，只保留五个支柱（Intent + Memory + Identity +
 *     Orchestration + Reflection，合称 IMIOR Pattern）。其余业务领域全部
 *     归入对应的 Domain OS（见 D2/D7），不再往 Core 里堆。
 *
 *     Intent ── 顶层意图路由
 *       职责：接进来的输入先判断属于哪个 Domain（Task？Inventory？
 *       Insight？），再分派下去——不是判断"具体要做什么"（那是各 Domain
 *       OS 自己 Request/Intent Engine 的工作，见 D6/D8），只判断"归哪个
 *       Domain 管"。
 *       Input Adapter：Intent 的输入口不限于文字。Telegram 文字消息是
 *       现在唯一实现的 Adapter；Voice（语音→STT→文字）是将来可能加的
 *       第二个 Adapter；Image / PDF / Email 也都是同一类东西——全部
 *       转成统一的文字/结构化输入后，走同一套 Intent Router，不会因为
 *       多一种输入方式就多建一个 OS（不然以后会冒出 Keyboard OS /
 *       Voice OS / Image OS / PDF OS / Email OS，这些本质都只是
 *       Adapter，不是业务领域）。
 *       现状实现：04_Main.gs 的 _handleMessage_ 分派逻辑
 *       （/help 全局短路 → handleTaskIntent → handleInventoryIntent →
 *       handleInsightIntent → 兜底），本质就是一个还没正式命名的
 *       Intent Router，目前只接了 Telegram 这一个 Adapter。Domain OS
 *       逐步拆出去之后，这里会演变成更正式的路由层，把消息转给各 OS
 *       自己的 Request/Intent Engine，而不是像现在这样直接调具体
 *       handler 函数。
 *
 *     Memory ── 原始记忆聚合
 *       职责：只读 EventBus，把 Events 聚合成事实性的记忆（完成率、
 *       风险标记之类），不做判断/不给建议——给建议是 Reflection 的事。
 *       现状实现：93_MemoryEngine.gs
 *
 *     Identity ── 身份与去重
 *       职责：确定性身份生成 + 去重判断 + 幂等创建，是 P1-P3（EventBus
 *       唯一写入/真相不可变/状态从事件派生）能够安全落地的地基。
 *       现状实现：07_IdentityEngine.gs（身份生成）+ 08_DeduplicationEngine.gs
 *       （去重判断）+ 09_IdempotencyManager.gs（幂等创建入口）
 *       这三个文件对外暴露的 generateXxxIdentity / findExistingXxx /
 *       createXxxIfNotExists 这套模式，是 Productivity OS / Reminder
 *       OS / Property OS 这类"归 Core 治理"的 Domain OS 未来复用的公开
 *       API——但不包括 D2 里的 External System（Procurement/Inventory/
 *       Rider OS 都是独立身份系统，不接 Core 的 Identity Registry，
 *       这也是 2026-07-03 把它们改判为 External System 的直接原因
 *       之一：既然不共享 Identity，就不该假装它们是"归 Core 治理"的
 *       Domain OS）。
 *       ⚠️ 已知遗留问题：07_StateEngine_TaskMetrics.txt 物理文件名跟
 *       File_Map 登记的 07_IdentityEngine.gs 不一致（2026-07-01 外部审计
 *       LOW RISK 2，已知未处理），仍按原计划列在 00_Project_State.gs
 *       的"下一步"里，不在这次架构声明范围内。
 *
 *     Orchestration ── 事件写入与读取编排
 *       职责：P1（EventBus 唯一写入者）和 P3（状态从事件派生）的直接
 *       实现层——事件写入、增量投影、全量重建、统一查询入口。
 *       现状实现：02_EventBus.gs（唯一写入口）+ 10_ProjectionEngine.gs
 *       （增量投影）+ 11_ProjectionRebuilder.gs（全量重建，手动运维）+
 *       12_QueryEngine.gs（统一读入口）
 *
 *     Reflection ── 模式分析与建议
 *       职责：在 Memory 的原始聚合之上做更高阶的模式分析、产出建议、
 *       路由洞察——是 P4"AI 只建议"的具体承载层。
 *       现状实现：94_BehaviorEngine.gs（依赖 93，算延迟风险/最佳时段）+
 *       95_SuggestionEngine.gs（依赖 93+94，产出文字建议，不写 Events）+
 *       96_InsightRouter.gs（93/94/95 的 Telegram 入口）
 *
 * D4. Shared Services
 *
 *     Level 1 Platform Standard 的第三个组成部分：不属于 IMIOR 五支柱，
 *     但 Core 和所有 Domain OS 都可能直接调用的底层公共设施，不重新
 *     发明，也不用"归到哪个支柱"这个问题硬套：
 *       01_SecureConfig.gs（密钥管理）
 *       03_Output.gs（Telegram 唯一发送口，P5）
 *       05_SheetUtils.gs（Sheet 读写工具，upsertRowByKey_ /
 *         batchUpsertRowsByKey_ 等）
 *       15_Setup.gs（部署/触发器管理）
 *     判断标准：如果一个工具函数不关心"这是哪个 Domain 的数据"，纯粹是
 *     通用能力（发消息/读写Sheet/管密钥/管触发器），就属于 Shared
 *     Services，不属于某个 Domain OS 私有，也不算 IMIOR 的业务能力。
 *
 * D5. Cross-domain Bridge
 *
 *     Level 1 Platform Standard 的第四个组成部分：任何"跨系统/跨 Domain
 *     OS"的集成点，统一走同一套 Bridge 约定，不各自发明协议。分两种
 *     具体形态：
 *
 *     (1) Core ↔ External System（D2 的 Rider OS / Procurement OS /
 *         Inventory OS）：
 *       - 命名：NN_XxxConnector.gs（沿用 C1 的文件命名规则），
 *         80_RiderConnector.gs 是现成范例。
 *       - 方向：默认单向只读（外部系统 Sheet/API → Core Task/Event），
 *         需要双向同步必须在 Connector 文件头部显式声明理由。
 *       - 边界：Connector 只做"读取+格式转换+调用本侧幂等创建入口
 *         （如 createTaskIfNotExists）"，不代替对方系统做决策，也不
 *         把本侧的 Identity Registry / EventBus 暴露给对方系统直接写。
 *
 *     (2) Domain OS ↔ Domain OS（现在就有一个具体案例：Reminder OS，
 *         见 D2）：
 *       - Productivity OS（未来 Property/Finance/Vehicle OS 也一样）
 *         需要提醒时，不会自己实现一套提醒逻辑，也不会直接写 Reminder
 *         OS 的 Sheet/Projection——而是调用 Reminder OS 自己暴露的
 *         Request/Intent Engine 入口（比如"创建一条提醒规则"），让
 *         Reminder OS 走它自己完整的 Level 2 生命周期去处理（自己的
 *         Planner/Decision/必要时的 Confirmation/Execution/Event）。
 *       - 换句话说：跨 Domain OS 永远是"调用对方的入口，让对方走完自己
 *         的流程"，不是"直接改对方的数据"——这样每个 Domain OS 的
 *         Level 2 生命周期完整性才不会被跨域调用破坏。
 *
 *     (3)【2026-07-08 新增】Core ↔ 受本 Constitution 治理的 Domain OS
 *         （目前两个具体案例：Productivity OS、Reminder OS）：统一
 *         Connector Layer：
 *       - 位置：08_Connector/ 目录（GAS 无真实文件夹，贴代码时按
 *         08_Connector/README.gs 的说明处理；跟 (1)(2) 的
 *         NN_XxxConnector.gs 单文件形态不同，这里是"共用基础设施 3 个
 *         文件（ConnectorResponse/ConnectorTypes/ConnectorRegistry）+
 *         每个 Domain OS 一个 Connector 文件"）。
 *       - 跟 (1)(2) 的关系：(1)(2) 回答的是"该不该建 Bridge、Bridge
 *         该长什么形态"，仍然成立；本条是在此基础上把 Core 侧"怎么
 *         调用"进一步标准化——每个受治理 Domain OS 应该有且仅有一个
 *         实现 Standard Connector Interface 的 Connector，统一通过
 *         ConnectorRegistry 查找/调用，不再像 80_RiderConnector.gs
 *         （(1) 类 External System Connector，命名/方法/返回形状都是
 *         各自随意设计的）那样每个 Connector 一套风格。
 *       - Standard Connector Interface：create/update/delete/get/
 *         list/search/execute/health/capabilities 九个方法，统一返回
 *         形状固定的响应 DTO（success/data/error/message/metadata/
 *         timestamp/duration/traceId），四类标准错误分类
 *         （VALIDATION_ERROR/BUSINESS_ERROR/SYSTEM_ERROR/UNKNOWN_ERROR）。
 *         完整定义见 08_Connector/README.gs、ConnectorTypes.gs、
 *         ConnectorResponse.gs。
 *       - 范围：目前覆盖 Productivity OS（ProductivityConnector.gs，
 *         走 Library 调用）和 Reminder OS（ReminderConnector.gs，
 *         Reminder OS 没有 Library/webhook——见 D2——只能读同一张共享
 *         Sheet，因此该 Connector 是只读的，且写操作会诚实返回
 *         BUSINESS_ERROR 而不是假装支持，理由见 08_Connector/README.gs
 *         五）。80_RiderConnector.gs 等 (1) 类 External System
 *         Connector 暂不强制迁移到这套统一接口，是否/何时迁移属于
 *         后续决定（见 00_Project_State.gs 下一步）。
 *       - 迁移策略：04_Main.gs / 80_RiderConnector.gs /
 *         21_InventoryModule.gs 现有直接调 ProductivityOS Library 的
 *         代码本次未改动，继续正常运作——遵循 P6.2"现有系统不需要为了
 *         新标准立即重构，走适配器模式逐步迁移"的既有原则。新写的、
 *         需要跟 Productivity OS/Reminder OS 交互的代码，应该优先走
 *         ConnectorRegistry，不再直接调 ProductivityOS.xxx()。
 *       - 已确认的能力边界（落地时发现的既有事实，不是本条新引入的
 *         限制）：Productivity OS 的 DeleteTask 映射为 cancelTask
 *         （软删除，事件溯源架构没有硬删除）；ReopenTask 不支持
 *         （status 不在 updateTask 的字段白名单内）；Reminder OS 的
 *         CreateReminder/UpdateReminder/DeleteReminder/
 *         CompleteReminder/DismissReminder/SnoozeReminder/
 *         TriggerReminder 在当前版本下均无对应后端能力——Connector
 *         对这些操作诚实返回 BUSINESS_ERROR + 具体原因，不代为实现
 *         （代为实现等于在 Core 里发明第二套业务逻辑，违反本文件 C5）。
 *
 *     Domain OS 内部自己的 Bridge（D6 Intelligence.Bridge）是这条在
 *     OS 内部的具体实现位置（调用对方入口的代码就住在这一层），不是
 *     另一套规则。
 *
 * D6. Domain OS Blueprint
 *
 *     所有 Domain OS（D2 里"归 JARVIS Platform 治理"的那些，不包括
 *     External System）按同一个结构模板组织，分三层：
 *
 *       Domain OS Blueprint
 *       │
 *       ├── Foundation（地基）
 *       │     ├── Governance   本 OS 的职责边界/依赖声明。规模小的时候
 *       │     │                 可以只在 00_File_Map.gs 里占一个章节；
 *       │     │                 规模大到需要独立维护时，参照 Core 的
 *       │     │                 C6 模式，给这个 OS 单独建自己的
 *       │     │                 Constitution / State / File Map 三件套
 *       │     │                 （不强制一开始就建，长大了再拆）。
 *       │     └── Identity     复用 Core.Identity（D3）暴露的公开 API，
 *       │                       不自己重新发明身份生成/去重逻辑。
 *       │
 *       ├── Runtime（运行时）
 *       │     └── Domain Pattern = P6.2 Level 2 Domain OS Standard
 *       │           （Request→Planner→Decision→Confirmation→Execution→
 *       │           Event→Projection→Insights→Bridge），不在这里重复
 *       │           定义，直接引用 P6.2。
 *       │
 *       └── Intelligence（智能层）
 *             ├── Knowledge   该 OS 特有的参考/查找数据（比如未来
 *             │                Property OS 的租约条款库）——是域内事实性
 *             │                参考数据，不是分析结果。
 *             ├── Insights    该 OS 的 Reflection 具象化——复杂到需要自己

 *             │                简单的话直接复用 Core.Reflection（D3）。
 *             └── Bridge      跨系统集成层，规则见 D5，不重复定义。
 *
 *     Runtime.Domain Pattern 的完整生命周期定义、豁免条款（现有系统不用
 *     立即重构/新系统必须完整遵循）见 P6.2，本条不重复。

/**
 * A1. 系统数据流向（不可更改）
 *
 *   User Input
 *     → EventBus
 *     → TruthEngine
 *     → EventSubscribers
 *       → ProductivityModule（任务相关事件）
 *       → ReminderEngine（监控任务，决定是否提醒）
 *       → RiderConnector（Rider OS 事件 → 任务，只读转换层）
 *       → InventoryModule（库存事件 → 任务）
 *     → Output（Telegram）
 *
 * A2. 模块之间不允许互相直接调用对方的内部状态存储，
 *     只能通过：
 *       (a) EventBus.publish() 发事件，或
 *       (b) 调用对方暴露的公开派生函数（如 getPendingTasks()）
 *
 * A3. ModuleRegistry 架构保持「领域无关」（domain-agnostic）
 *     新模块要尽量复用 Task 数据模型，而不是给每个领域（保养/库存/提醒）
 *     单独发明一套状态结构。
 *
 * A4. 任何模块要发 Telegram 消息，必须经过 Output.gs（见 P5）
 */


// ============================================================
// 四、Coding Rules
// ============================================================

/**
 * C1. 文件命名：NN_ModuleName.gs，NN 是两位数字前缀，决定模块在系统里的层级
 *     （例如 00 = 治理文档，07 = StateEngine，20+ = 业务模块，9x = 引擎/调度类）
 *
 * C2. 每个模块的「写」操作必须是 EventBus.publish(type, payload, chatId, source)，
 *     不允许出现 sheet.appendRow() 写状态数据（治理文档/日志类例外）
 *
 * C3. 「读」操作走派生函数（deriveXxxState_()），对外暴露的查询函数
 *     （如 getPendingTasks）内部也是调派生函数，不缓存结果到全局变量持久化
 *
 * C4. 每个 Task 必须有 category（不能是 undefined / 空字符串）
 *
 * C5. Reminder / Inventory 等监控类逻辑只允许存在一套实现，
 *     发现重复实现要合并，不要并存两套相似逻辑
 *
 * C6. 🔴🔴🔴 同步规则（强制，不可省略） 🔴🔴🔴
 *     任何代码变更 —— 新增模块、修改架构、修 Bug、改数据模型、加字段 ——
 *     必须同步更新以下三份文件中【受影响的那一份或几份】：
 *
 *       - 00_Project_Constitution.gs  → 原则/架构/规则变了才更新
 *       - 00_Project_State.gs         → 几乎每次变更都要更新（已完成/进行中/Bug/下一步）
 *       - 00_File_Map.gs              → 新增/删除/重命名文件，或模块关系变了才更新
 *
 *     执行时机：写完代码的同一轮回复里就要同步，不允许「先写代码，
 *     下次再补文档」。新开窗口交接时，只需要贴这三份文件 + 当前要改的文件，
 *     不需要贴全部源码。
 *
 * C7. 派生视图表（Tasks / Inventory / ActiveTasks / ArchiveTasks 等）的读写
 *     必须用 05_SheetUtils.gs 的 getHeaderMap_() / upsertRowByKey_() /
 *     deleteRowByKey_() / batchUpsertRowsByKey_()，按表头名字定位列，
 *     不允许写死列号（如 sheet.getRange(row, 3)）—— 表头顺序/新增列
 *     不应该导致代码出错。
 *
 * C8.【2026-06-25新增，范围受限的例外】02_EventBus.gs 的 getAllEvents()
 *     允许维护一个「执行期内存缓存」_cachedEvents：只在同一次脚本执行
 *     （一次 doPost 或一次 Trigger 调用）的生命周期内存活，publish() 写入
 *     新事件时立即失效，不跨执行持久化。这是为了避免同一次执行里反复
 *     物理读取 Events 表（比如一次INVENTORY_USE内部要deriveInventory()
 *     三次）。
 *     这条不违反 C3「不缓存结果到全局变量持久化」的精神：C3 针对的是
 *     deriveTaskState_/deriveInventory这类「派生函数」不允许引入跨执行的
 *     第二套持久状态（会和Events表脱钩、产生数据不一致）；而这里缓存的
 *     是EventBus读到的原始事件本身，且寿命≤一次执行、写入即失效，
 *     不构成独立于Events表的状态。
 *     除 EventBus.getAllEvents() 外，其余模块（包括 deriveTaskState_ /
 *     deriveInventory 等派生函数）依然禁止引入跨执行的全局缓存。
 *
 * C9.【2026-07-03 新增】新建 Domain OS / Engine 时必须遵循 D6 Blueprint +
 *     P6.2 的 UDP（Level 2 Domain OS Standard / Level 3 Engine Standard，
 *     按需选用），不能自创一套生命周期。如果某个场景确实不适合套用（比如
 *     一次性脚本/纯运维工具），要在文件头部写清楚不适用的理由，不能默不
 *     作声地跳过——跟 P6.2 Level 2"新系统必须完整遵循，不能自动豁免"
 *     是同一条精神。
 */
